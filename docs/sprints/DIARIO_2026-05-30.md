## Diagnóstico clave del día (SPRINT-WIZARD-FASES-FREEZE)

Cowork reportó el 2026-05-30: clic "En Cotización" desde el wizard del modal de OS-0057 congela el navegador (`document_idle waited 45000ms`, reproducible 2/2). Bloquea QA visual de DINERO-2.

**Diagnóstico del coordinator:** no es un bug del producto para usuarios reales. Cowork hizo el QA con Playwright MCP (`.playwright-mcp/` untracked en el árbol). El handler `handleClickFase` en `FaseStepper.tsx:168` hacía `window.confirm()` antes de avanzar la fase. **Playwright NO maneja dialogs nativos por default** — sin `page.on('dialog', dialog => dialog.accept())` instalado, el confirm queda esperando indefinidamente → `document_idle` nunca llega.

**Fix aplicado:** reemplazar `window.confirm` por Modal propio (mismo componente `Modal` que ya usaba el flujo de retroceso). Beneficio doble: (a) elimina el cuelgue en Playwright permitiendo QA automatizado E2E, (b) mejora UX para usuarios reales (modal consistente con el resto del sistema).

**Cero regresión:** los 4 consumidores de `FaseStepper` (OrdenDetailModal, OrdenDetalle, OrdenCard, TecnicoVista) NO pasan `onCambioFase` — todos usan el `ejecutarCambio` interno. TecnicoVista lo monta `readonly={true}` → handler no se dispara. Comportamiento idéntico.

**Deuda follow-up identificada (NO scope):** `FaseStepper.tsx:71-77` reconstruye `historialFases` con `h.timestamp instanceof Date ? h.timestamp : new Date()`. Si `h.timestamp` viene como Firestore Timestamp (caso normal post-read), `instanceof Date` es false → reemplaza con `new Date()` (hoy), perdiendo el timestamp histórico real. Bug latente, no causa freeze, sprint propio con QA del historial de fases.
