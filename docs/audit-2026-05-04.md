# Audit completo — 2026-05-04

5 agentes especializados (security, dinero, integridad, state, error handling) revisaron en paralelo. **46 hallazgos brutos** reducidos a **30 hallazgos reales** después de deduplicar y verificar falsos positivos.

## Falsos positivos descartados (verificados manualmente)

1. **`aplicarCuota()` doble-aplicación si retry** (`prestamos.service.ts:150-191`). Agente claimed la idempotencia falla en retry. **Falso**: la operación está dentro de `runTransaction` con guard `histRaw.some(h => h.liquidacionId === liquidacionId)`. Firestore garantiza atomicidad read+write. Idempotencia correcta.

2. **`formatMonedaPrecisa` NaN propagation** (`utils/index.ts:60`). Agente claimed bug por strings con coma. **Falso**: la función espera `number`, recibir string es bug del caller. Función correcta.

3. **`rangoQuincena` Q1 febrero off-by-one** (`comisiones.ts:188-192`). Agente claimed Q1 febrero rompe. **Falso**: `m-2` para febrero da `0` que es enero (correcto: Q1 feb = 30-31 enero + 1-14 feb). El cálculo está bien. Comentario de fix #77 está claro.

## CRÍTICOS (verificados, fix prioritario)

### C1 — Phone normalization acepta códigos internacionales NO-RD
- **Archivo**: `src/services/clientes.service.ts:7-13`
- **Bug**: `replace(/\D/g, '')` mata el `+`. Un número `+56-8095551234` (Chile) se normaliza a `8095551234` y matchea con cliente RD legítimo. La condición `length === 11 && startsWith('1')` solo cubre +1 explícito.
- **Impacto**: clientes internacionales fantasma se mergean con clientes RD reales. Pierde historial. Whatsapp links incorrectos.
- **Fix**: rechazar si `soloDigitos.length > 11` o si `length === 11` pero NO `startsWith('1')`. Retornar `''` o lanzar.

### C2 — App Check soft enforcement (sin bloqueo real)
- **Archivos**: `api/_lib/firebaseAdmin.ts:75-90`, todos los endpoints públicos
- **Bug**: fase A loguea `app_check.ok` pero NO bloquea requests sin token. Diseño intencional (sprint C3 fase A) pero pendiente cierre.
- **Impacto**: bots pueden hacer fuzzing de tokens portal-cliente y garantía sin App Check.
- **Fix**: ejecutar fase B (~30 min) — `if (!appCheckResult.ok) return res.status(401)` en handlers públicos. Ya está en cola en `PROMPT_AUDIT_C3_APPCHECK.md`.

### C3 — AppContext synthesized administrador profile sin persist
- **Archivo**: `src/context/AppContext.tsx` (3-step cascade documentada en CLAUDE.md)
- **Bug**: si user de Firebase Auth NO tiene doc en `usuarios/{uid}` ni en `personal where email==`, se crea un perfil `administrador` en memoria. Documentado como "demo mode".
- **Impacto**: cualquier usuario autenticado en Firebase Auth con email no registrado en personal → acceso admin sin auditoría.
- **Fix**: eliminar el demo fallback. Si no hay perfil real → mostrar error "Perfil no encontrado, contactá admin". Romperá demo mode pero es trade-off correcto en producción.

### C4 — Avance marcado como descontado sin verificar aplicación
- **Archivo**: `src/services/nomina.service.ts:343-345`
- **Bug**: `cerrarLiquidacion()` marca todos los avances como `descontado=true` con `Promise.all` paralelo. Si `updateDoc` de un avance falla pero el catch lo silencia, queda `descontado=true` sin haberse aplicado el descuento real en nómina.
- **Impacto**: empleado recibe extra (descuento no aplicado) pero el avance figura saldado. Pérdida silenciosa.
- **Fix**: wrap individual try/catch que lance al primer fallo. NO permitir cerrar liquidación si algún avance falla. Idealmente transactionalizar junto con el update de la liquidación.

### C5 — Denormalización N>1 técnicos puede saltarse
- **Archivo**: `src/components/facturacion-pendiente/ProcesarFacturacionModal.tsx:482-504`
- **Bug**: el caller solo denormaliza si `result.comisiones.length > 0`. Si todas las comisiones son 0% pero hubo `preservadasPorLiquidacion > 0` (cleanup huérfanas), salta la denormalización. Factura queda con `comisionTecnicoMonto = null` aunque comisiones existen en colección.
- **Impacto**: reportes de facturas muestran "—" aunque comisión sí está en la colección `comisiones`. Inconsistencia entre dos vistas del mismo dato.
- **Fix**: denormalizar siempre que haya actividad (length > 0 OR preservadasPorLiquidacion > 0 OR totalAgregado > 0). Wrap en try/catch defensivo.

## ALTOS (verificar antes de fix)

### A1 — Coords sin validación de rango en parseOrden
- **Archivo**: `src/utils/index.ts:642-643, 733-734`
- **Bug**: `typeof ic.lat === 'number'` no valida rango `[-90,90]`. Coords inválidas (ej: `lat=500`) se persisten silenciosamente.
- **Impacto**: pins en ubicaciones imposibles, `distanciaCliente` con NaN.
- **Fix**: post-parse `Math.abs(lat) <= 90 && Math.abs(lng) <= 180` o retornar `undefined`.

### A2 — `parseFirestoreDate` retorna Invalid Date para strings inválidos
- **Archivo**: `src/utils/index.ts:504-512`
- **Bug**: `if (typeof val === 'string') return new Date(val)` acepta cualquier string. Strings malformados producen `Invalid Date` que propaga errors silenciosos.
- **Impacto**: `differenceInDays`, `format`, `getTime` rompen o devuelven valores tontos. Alertas con timestamps inválidos.
- **Fix**: `const d = new Date(val); return isNaN(d.getTime()) ? null : d;`

### A3 — Dashboard listener de comisiones no llama `checkLoaded()`
- **Archivo**: `src/pages/Dashboard.tsx:158-178`
- **Bug**: 5 de 6 listeners llaman `checkLoaded()` en su callback. El de comisiones (línea 158) NO. Si comisiones está vacía o tarda, `loadedCount` nunca llega a 6 y `setLoading(false)` nunca dispara.
- **Impacto**: Dashboard queda en "Cargando..." indefinidamente cuando no hay comisiones pendientes.
- **Fix**: agregar `checkLoaded()` al callback de `unsubComisiones`.

### A4 — `ProcesarFacturacionModal` localStorage borrador sin guard "ya restauré"
- **Archivo**: `src/components/facturacion-pendiente/ProcesarFacturacionModal.tsx:112-200`
- **Bug**: viola la regla nueva de CLAUDE.md ("effects con localStorage requieren guard yaRestauradoRef"). Sin el guard, otro effect puede pisar el borrador antes de que el usuario tenga chance de restaurarlo.
- **Impacto**: borrador recuperado y luego pisado silenciosamente.
- **Fix**: agregar ref `yaRestauradoRef` o condicionar el save a `borradorEncontrado === null`. Patrón ya en CLAUDE.md gotcha.

### A5 — Sidebar abre 7+ listeners sin filtrar la colección
- **Archivo**: `src/components/Sidebar.tsx:75-155`
- **Bug**: `onSnapshot(collection(db, 'ordenes_servicio'))` SIN where en 3 lugares distintos. Cada cambio dispara 3 snaps que leen la colección entera.
- **Impacto**: con >10k órdenes, picos de cuota Firestore reads → costo + latencia. Sidebar es permanente, corre todo el tiempo.
- **Fix**: usar índices compuestos con `where()` puntuales, o consolidar conteos en un doc `stats` que el Sidebar escucha.

### A6 — Rate limiting ausente en endpoints públicos
- **Archivos**: `api/portal-cliente/[token].ts`, `api/garantia/[token].ts`, `api/ai/chat.ts`
- **Bug**: ningún rate limiting. Bot puede hacer miles de requests/seg.
- **Impacto**: DoS de cuota Firestore, fuzzing de UUIDs de órdenes (enumeration), spam de reclamos garantía falsos, abuso de IA endpoints.
- **Fix**: rate limiting por IP (Vercel Edge Middleware o Upstash Redis). 10 req/min para portal-cliente, 5 req/min para ai/chat.

### A7 — Error messages exponen detalles internos
- **Archivos**: `api/ai/chat.ts:337,525`, `api/gps/ubicacion.ts:220`, `api/portal-cliente/[token].ts:232`
- **Bug**: `err.message.substring(0, 300)` retornado al cliente. Expone arquitectura (Firestore, colecciones consultadas).
- **Impacto**: facilita reconocimiento del sistema a atacantes.
- **Fix**: en producción, retornar genérico "Error interno". Loguear el detalle solo server-side.

### A8 — Logs con datos sensibles (tokens, UIDs, emails)
- **Archivos**: varios endpoints `api/`
- **Bug**: `console.log(JSON.stringify({uid, token: token.substring(0,8)...}))`. Tokens parciales y UIDs en logs.
- **Impacto**: si Vercel logs se exportan a un servicio terceros, exposición de identificadores.
- **Fix**: nunca loguear tokens/UIDs/emails. Usar request UUID interno para tracing.

### A9 — Técnico huérfano genera comisión 0% sin warning visual
- **Archivo**: `src/utils/comisiones.ts:481-493`
- **Bug**: si item tiene `tecnicoId` que NO existe en `personal/`, retorna `{nombre:'', porcentaje:0}`. Crea comisión silenciosa con monto 0.
- **Impacto**: técnico fantasma queda en historial sin trazabilidad clara. Admin no ve warning.
- **Fix**: si `getTecnico` retorna porcentaje 0 → skip (no crear comisión). UI muestra warning "items sin técnico válido" antes de emitir.

### A10 — Cliente B2B sin tipo configurado emite con precio Detalle silenciosamente
- **Archivo**: `src/components/facturacion-pendiente/ProcesarFacturacionModal.tsx:323-324`
- **Bug**: si `cliente.tipo` viene undefined (legacy) → default `particular`. Modalidad por línea queda `detalle`. Override por modal NO se audita si admin no toca el toggle.
- **Impacto**: facturas B2B emitidas a precio detalle (más caro) → cliente reclama → no hay audit trail.
- **Fix**: badge visual "Cliente sin tipo definido — verificar" en el modal cuando `tipo` es default. Decisión G2 que ya pasó a cleanup.

### A11 — Si `registrarComisionPorFactura` falla, comisión queda huérfana
- **Archivo**: `src/utils/comisiones.ts:840-841`
- **Bug**: catch loguea pero retorna parcial (`comisionId=null`). Caller denormaliza factura igualmente.
- **Impacto**: factura muestra "comisión RD$X" pero no existe doc en colección. Técnico no cobra.
- **Fix**: relanzar error si addDoc falla. UI debe reintentar o mostrar estado "no persistida".

### A12 — Préstamos múltiples activos: fallo parcial en aplicación de cuotas
- **Archivo**: `src/services/nomina.service.ts:210-219`
- **Bug**: si empleado tiene 2 préstamos activos y `aplicarCuota` falla en uno de los dos en `cerrarLiquidacion`, el otro se aplica de todas formas. Discrepancia entre `totalCuotasPrestamos` mostrado vs descuentos efectivos.
- **Impacto**: empleado descontado por menos cuotas de las contabilizadas → cobra extra. Solo si hay error transient.
- **Fix**: secuencializar (no Promise.all) y abortar al primer fallo, o pre-validar préstamos existen antes de marcar cerrada.

### A13 — Citas.tsx fetch comisión original sin AbortController
- **Archivo**: `src/pages/Citas.tsx:153-192`
- **Bug**: usa flag `cancelado` local en lugar de AbortController. Si user abre/cierra modal rápido, closure anterior puede llamar setState con datos viejos.
- **Impacto**: setState en componente no-mountado, warning en console, posible UI inconsistente.
- **Fix**: AbortController como `useAsistenteIAChat` (sprint 5).

### A14 — `JSON.parse` de localStorage sin try/catch en lectura de borradores
- **Archivos**: varios componentes con borradores localStorage
- **Bug**: si JSON está malformado (ej: usuario cambió a otra versión que escribió shape distinto), `JSON.parse` tira y rompe el componente.
- **Impacto**: pantalla blanca, usuario pierde acceso a la feature.
- **Fix**: `try { JSON.parse(...) } catch { return null; }`. Patrón estándar.
